# Axioms

Foundational beliefs that RootSpec is built on. These cannot be derived from the framework — they are the framework's reason for existing.

The framework specification (`00.FRAMEWORK.md`), skills, and product decisions are downstream of these axioms. If a framework rule contradicts an axiom, the axiom wins.

---

## 1. Philosophy guides implementation, never vice versa

Software degrades when implementation decisions reshape product intent. The direction of influence must be enforced structurally — not by discipline alone — so that foundational intent remains stable as implementation evolves.

## 2. Every detail traces to a user need

A feature, parameter, or system that cannot trace its existence back to a user need through a design pillar to a mission has no justification. Traceability is not documentation overhead — it is the mechanism that prevents scope drift and keeps the product coherent.

## 3. Hierarchy prevents circular dependency

When concerns at different abstraction levels reference each other freely, changes propagate unpredictably. Strict dependency inversion — where each layer can only reference layers above it — eliminates circular dependencies and makes change impact directional and predictable.

## 4. Validation requires structured interpretation

Without a structured interpretation layer between human intent and implementation, there is no way to verify that the output reflects the intent. This is true whether the output is hand-written, generated, or anything in between. The value is in validation and proof, not production.

## 5. Specification is extraction, not invention

The product vision already exists in the mind of its creator. The framework's job is to extract and formalize that latent vision through structured dialogue — not to impose a vision or generate one. The interview is the mechanism; the creator is the source of truth.

## 6. Single source of truth per concern

Each concern lives at exactly one level of abstraction. Duplication across levels creates drift — when two places define the same thing, they inevitably diverge, and the system loses coherence.

## 7. Implicit intent drifts — the spec must externalize it

Humans hold intent implicitly — ethics, values, experience, conventions, cultural norms. At each step from idea to implementation, this implicit understanding degrades. Much of what makes a product coherent lives in the creator's unarticulated knowledge. The spec surfaces and embeds this within the project itself, where any agent — human or AI — can reference it as needed, so that output doesn't gradually diverge from the source.

## 8. Natural language is the specification medium

The spec is written in natural language because that is how understanding transfers. LLMs were trained on human text; the implicit understandings embedded in that training are activated by natural language context. The spec leverages this — it is not a configuration format, it is a vehicle for meaning.

## 9. Structure directs attention

Neither humans nor LLMs can consider everything at once. The hierarchical structure puts the right context in scope at the right time — philosophy when making design decisions, system specs when implementing, user stories when validating. The spec is not documentation; it is an attention-directing mechanism.
